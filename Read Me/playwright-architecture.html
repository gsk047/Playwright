01 — Overview

What is Playwright?

Playwright is an open-source browser automation framework by Microsoft. It lets you control real browsers (Chromium, Firefox, WebKit) programmatically — primarily for end-to-end testing, but also for scraping, screenshots, and automation.

Core Library

Playwright

The core API. Provides browser, page, and context objects. Speaks to browsers via Chrome DevTools Protocol (CDP) and a custom protocol for Firefox/WebKit.

Test Runner

Playwright Test

A full-featured test runner built on top of Playwright. Adds fixtures, parallelism, retries, reporters, and a rich assertion API. Replaces Jest/Mocha for E2E tests.

Multi-Browser

Cross-Browser

Chromium (Chrome/Edge), Firefox, and WebKit (Safari) — all with a single unified API. Write once, run everywhere.

02 — Core Architecture

The Layered Architecture

Playwright is built in layers. Each layer has a specific responsibility and communicates strictly top-down.

🧑‍💻

User Code / Test Scripts

Your TypeScript/JavaScript/Python/Java/C# code. You write tests using the Playwright API — page.goto(), locator.click(), expect(), etc.

↓ JSON-RPC Messages ↓
🔌

Playwright Node.js Server (playwright-core)

The "brain." Maintains browser state, implements retries, enforces auto-waiting, and serializes/deserializes browser commands. Written in TypeScript, runs in Node.js.

↓ WebSocket / CDP Protocol ↓
🖥️

Browser Process + Driver

Actual browser binary (Chromium, Firefox, WebKit). Each browser has its own patched build. Playwright uses CDP for Chromium and a custom protocol for Firefox/WebKit.

↓ Renders ↓
🌐

Web Page (Renderer Process)

The actual page being tested, running in an isolated browser context. Playwright can inject scripts, intercept network, and inspect the DOM here.

03 — Core Objects

The 4 Core Objects

Every Playwright operation revolves around four objects arranged in a strict hierarchy.

Level 1

🌐 Browser

Represents one browser instance (e.g., Chromium). You launch it once and use it to create contexts. One browser can have many contexts.

Level 2

📦 BrowserContext

Like an incognito window. Fully isolated — separate cookies, storage, cache. Great for running tests as different users in parallel.

Level 3

📄 Page

A single browser tab. You interact with the page: navigate, click, type, screenshot. One context can have multiple pages (tabs).

Level 4

🎯 Locator

A smart selector for DOM elements. Unlike querySelector, Locators auto-wait for elements to be visible, stable, and interactable before acting.

hierarchy.ts
// The object hierarchy in code const browser = await chromium.launch(); // Level 1 const context = await browser.newContext(); // Level 2 — isolated const page = await context.newPage(); // Level 3 — single tab const button = page.locator('button[type="submit"]'); // Level 4 — smart element await page.goto('https://example.com'); await button.click(); // auto-waits for visibility + stability first
04 — Key Mechanism

Auto-Waiting: The Killer Feature

The #1 cause of flaky tests is race conditions — acting on an element before it's ready. Playwright solves this with built-in auto-waiting on every action.

How Auto-Waiting Works

Before every action (click, fill, check), Playwright verifies a series of "actionability conditions" on the target element. Only when all pass does the action execute — no manual waitForSelector() calls needed.

05 — Playwright Test

Playwright Test Architecture

Playwright Test is a full test runner — not just a wrapper. It has its own architecture built on fixtures, worker processes, and a project system.

Dependency Injection

⚙️ Fixtures

The foundation of Playwright Test. Fixtures provide setup/teardown for each test. Built-ins: page, browser, context, request. You compose and extend them freely.

Parallelism

👷 Worker Processes

Tests run in parallel across isolated Node.js Worker processes (not threads). Each worker has its own browser instance. Workers scale to your CPU cores automatically.

Multi-Config

📁 Projects

"Projects" run the same tests across multiple browsers or configs at once. One file → Chromium + Firefox + WebKit + Mobile, all in one run.

playwright.config.ts
export default defineConfig({ testDir: './tests', fullyParallel: true, // all tests run in parallel workers: process.env.CI ? 1 : 4, // 4 workers locally, 1 in CI retries: process.env.CI ? 2 : 0, // auto-retry flaky tests in CI reporter: 'html', // beautiful HTML report projects: [ { name: 'chromium', use: { ...devices['Desktop Chrome'] } }, { name: 'firefox', use: { ...devices['Desktop Firefox'] } }, { name: 'webkit', use: { ...devices['Desktop Safari'] } }, { name: 'Mobile', use: { ...devices['iPhone 14'] } }, ], });
06 — Fixtures Deep Dive

The Fixture System

Fixtures are Playwright Test's superpower. Think of them like dependency injection — they set up preconditions and inject them into your tests without boilerplate.

fixtures.ts
// Extend base test with a "loggedInPage" fixture const test = base.extend<{ loggedInPage: Page }>({ loggedInPage: async ({ page }, use) => { // SETUP — runs before each test await page.goto('/login'); await page.fill('#email', 'user@example.com'); await page.click('button[type=submit]'); await use(page); // ← hand control to the test // TEARDOWN — runs automatically after each test await page.evaluate(() => localStorage.clear()); } }); // Clean usage — no beforeEach/afterEach needed! test('user sees dashboard', async ({ loggedInPage }) => { await loggedInPage.goto('/dashboard'); await expect(loggedInPage).toHaveTitle('Dashboard'); });

Two Fixture Scopes

test scope (default): Fresh instance per test — isolation guaranteed.  |  worker scope: Shared across all tests in the same worker — use for expensive operations like DB seeding or auth setup that only needs to happen once.

Remote Execution

Playwright can connect to a remote browser via browserType.connect(wsEndpoint). This is how cloud services (BrowserStack, Sauce Labs) and Playwright's Docker image work — the Node server connects over the network to a browser running elsewhere.

08 — Advanced Features

Network, Context & Debugging

Network

Route Interception

Intercept any HTTP/HTTPS request. Mock API responses, simulate offline, abort requests. No proxy setup — works at context level.

Storage

Storage State

Save and restore cookies + localStorage via storageState. Authenticate once, reuse state across your whole suite. Eliminates login overhead.

Devices

Device Emulation

Emulate any mobile device with correct viewport, user agent, touch, and pixel ratio. 100+ preset devices included out of the box.

Debugging

Trace Viewer

Captures screenshots, DOM snapshots, network logs, and console output for every test. Replay failures step-by-step — no more "it only fails in CI."
